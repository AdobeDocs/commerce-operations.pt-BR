# Notas de versão do Magento Open Source 2.4.0

O Magento Open Source 2.4.0 apresenta suporte para PHP 7.4, Elasticsearch 7.6.x e MySQL 8.0. Alterações substanciais de segurança incluem a ativação da autenticação de dois fatores no Admin por padrão.

**Com esta versão, a integração do método Authorize.Net foi removida do código principal. Os comerciantes devem migrar para a extensão oficial disponível no Commerce Marketplace**.

>[!NOTE]
>
>**integração de Braintree payment**: Antes da versão 2.4.0, era recomendável que os comerciantes instalassem e configurassem a extensão de integração de pagamento Braintree oficial do Commerce Marketplace para substituir a integração principal. Com esta versão (2.4.0), a extensão agora é incluída na versão. Os comerciantes devem seguir etapas adicionais para garantir que o Braintree funcione corretamente em uma implantação 2.4.0. Consulte [Braintree](https://docs.magento.com/user-guide/payment/braintree.html) para obter mais informações sobre como migrar para a versão 2.4.0.

Esta versão inclui todas as melhorias na qualidade principal que foram incluídas na versão 2.3.5-p1, mais de 100 novas correções no código principal e 30 aprimoramentos na segurança. Ele inclui a resolução de 226 problemas do GitHub por nossos membros da comunidade. Essas contribuições da comunidade variam de uma pequena limpeza do código principal a melhorias significativas no Inventory management e GraphQL.

>[!NOTE]
>
>Versões secundárias trazem melhorias substanciais no código. Antes de atualizar para a versão 2.4.0, confirme se o ambiente atende ao mínimo [requisitos técnicos de pilha](../../../installation/system-requirements.md).

## Aplicar `AC-3022.patch` continuar a oferecer à DHL como transportadora

A DHL introduziu o schema versão 6.2 e descontinuará o schema versão 6.0 em breve. Adobe Commerce 2.4.4 e versões anteriores que oferecem suporte à integração com a DHL somente a versão 6.0. Os comerciantes que implantam essas versões devem se aplicar `AC-3022.patch` logo que possível, continuar a oferecer à DHL como transportadora marítima. Consulte a [Aplicar um patch para continuar oferecendo a DHL como transportadora](https://support.magento.com/hc/en-us/articles/7707818131597-Apply-a-patch-to-continue-offering-DHL-as-shipping-carrier) Artigo da Base de conhecimento para obter informações sobre como baixar e instalar o patch.

## Patch de segurança disponível

Os comerciantes agora podem instalar correções de segurança sensíveis ao tempo sem aplicar as centenas de correções funcionais e aprimoramentos que uma versão trimestral completa (por exemplo, 2.3.5-p2) oferece. O Patch 2.3.5.2 (Pacote Composer 2.3.5-p2) é um patch de segurança que fornece correções para vulnerabilidades que foram identificadas em nossa versão trimestral anterior, 2.3.5-p1. Todos os hotfixes que foram aplicados à versão 2.3.5 estão incluídos neste patch de segurança. A *hotfix* fornece uma correção para uma versão lançada que soluciona um problema ou bug específico.)

Para obter informações gerais sobre patches de segurança, consulte [Introdução à nova versão do patch de segurança](https://community.magento.com/t5/Magento-DevBlog/Introducing-the-New-Security-Patch-Release/ba-p/141287).Para obter instruções sobre como baixar e aplicar patches de segurança (incluindo patch 2.3.5-p2), consulte [Instalação no local de início rápido](../../../installation/composer.md). Os patches de segurança incluem somente correções de erros de segurança, não os aprimoramentos de segurança adicionais incluídos no patch completo.

### Melhorias substanciais na segurança

Esta versão inclui mais de 30 correções de segurança e melhorias na segurança da plataforma.

#### Mais de 30 aprimoramentos de segurança que ajudam a fechar as vulnerabilidades de execução de código remoto (RCE) e script entre sites (XSS)

Até à data, não ocorreram ataques confirmados relacionados com estas questões. No entanto, certas vulnerabilidades podem ser exploradas para acessar informações do cliente ou tomar as sessões do administrador. A maioria desses problemas exige que um invasor obtenha acesso primeiro ao Administrador. Como resultado, lembramos que você deve tomar todas as medidas necessárias para proteger seu administrador, incluindo, entre outras, esses esforços:  de IP incluir na lista de permissões, [autenticação de dois fatores](https://devdocs.magento.com/guides/v2.4/security/two-factor-authentication.html), uso de uma VPN, uso de um local exclusivo em vez de `/admin`e boa higiene de senhas. Consulte [Atualizações de segurança disponíveis para o Magento](https://helpx.adobe.com/security/products/magento/apsb20-47.html) para uma discussão sobre estas questões fixas. para uma discussão sobre estas questões fixas.

#### Melhorias adicionais na segurança

* **Implementação do 2FA para contas de administração, contas de usuário da magento.com e acesso à Cloud SSH**

   * **Proteção da sua conta de administrador**. A autenticação de dois fatores (2FA) agora é necessária para o Administrador. Os usuários administradores devem primeiro configurar o 2FA antes de fazer logon no Administrador por meio da interface do usuário ou de uma API da Web. O 2FA é ativado por padrão. É altamente recomendável não desativar o módulo 2FA.  Essa etapa extra de autenticação dificulta o logon de usuários mal-intencionados no Admin sem autorização. Consulte [Autenticação de dois fatores (2FA)](https://devdocs.magento.com/guides/v2.4/security/two-factor-authentication.html). <!-- MC-22631-->

   * **Proteção de sua conta**. A Autenticação de dois fatores (2FA) fornece uma camada adicional e opcional de segurança para proteger melhor sua conta da magento.com de usuários não autorizados que talvez queiram usar sua conta de maneiras não desejadas. Consulte [Protegendo sua conta](https://docs.magento.com/user-guide/magento/magento-account-secure.html).

* **Proteção do acesso à nuvem SSH**. A infraestrutura de nuvem do Adobe on fornece a aplicação de autenticação de vários fatores (MFA) para gerenciar os requisitos de autenticação para o acesso SSH a ambientes da nuvem. A autenticação multifator para 2FA não é habilitada por padrão em um projeto.  O Adobe recomenda habilitar esse recurso. Entre em contato com o Suporte para obter assistência. Consulte [Habilitar autenticação multifator para acesso SSH](https://devdocs.magento.com/cloud/project/project-enable-mfa-enforcement.html).

* **O modo estrito do filtro de modelo agora está ativado por padrão**. Os componentes (incluindo páginas e blocos CMS) que usam o filtro de modelo no modo herdado podem ser vulneráveis à execução remota de código (RCE). Ativar o modo estrito por padrão garante que os ataques de RCE não possam ser ativados deliberadamente. <!-- MC-22982-->

* **A renderização de dados para provedores de dados da interface do usuário agora está desabilitada por padrão**. Isso remove a oportunidade de usuários mal-intencionados executarem JavaScript arbitrário. <!-- MC-17356-->

* **Novo `\Magento\Framework\Escaper` classe**. Esta classe é fornecida para `.phtml` modelos e as classes PHP responsáveis pela geração de HTML. Essa classe contém métodos de inicialização de HTML relevantes para vários contextos. O `$escaper` variável local está disponível dentro de `.phtml` e devem ser usados em vez de `$block->escape{method}`. Use `$escaper` em vez de `$block` como a utilização de `$block->escape{method}` foi substituída.

* **Suporte para o novo arquivo security.txt**. Esse arquivo é um arquivo padrão do setor no servidor que ajuda os pesquisadores de segurança a relatar possíveis problemas de segurança para os administradores do site.

* **Aprimoramentos no suporte à Política de segurança de conteúdo (CSP)**. `SecureHtmlRenderer` foi adicionado à Estrutura e está disponível em `.phtml` modelos na lista de permissões em linha `style` e `script` tags. Não são permitidos scripts e estilos integrados com a configuração padrão da CSP, que pode ser substituída por arquivos XML. <!-- MC-34433-->

### Atualizações da plataforma

As atualizações de plataforma a seguir ajudam a melhorar a segurança e o desempenho do site. As versões compatíveis de PHP e PHPUnit, Elasticsearch, MySQL e outras dependências estão listadas em [requisitos de pilha de tecnologia](../../../installation/system-requirements.md).

* **suporte ao PHP 7.4 introduzido e PHP 7.1 e 7.2 obsoleto**. 2.4.0 apresenta suporte para PHP 7.4.

* **Suporte para PHPUnit 9.x e descontinuação do PHPUnit 6.5**. O PHP 7.4 requer o uso da mais recente estrutura de teste PHPUnit, que é a PHPUnit 9.x. Os fornecedores de extensão do Commerce Marketplace devem confirmar que todas as novas versões de extensão são compatíveis com o PHP 7.4 e que todos os testes de unidade e integração foram configurados para serem executados com PHPUnit 9.

* **Suporte ao Elasticsearch 7.6.x**. Elasticsearch 7.6.x agora é o mecanismo de pesquisa de catálogo padrão para Adobe Commerce e Adobe Commerce. Não é possível instalar ou atualizar para a versão 2.4.0 sem também instalar o Elasticsearch 7.6.x. O código Elasticsearch versão 2.x foi removido. As versões 5.x e 6.x do Elasticsearch foram substituídas e não são mais compatíveis. Consulte [Pré-requisitos do mecanismo de pesquisa](../../../installation/prerequisites/search-engine/overview.md).

* **Suporte para MySQL 8.0**. A versão 2.4.x é compatível com o MySQL 8.x. (2.4.0 foi testado com o MySQL 8.0.20.) Os comerciantes são incentivados a migrar suas implantações para o MySQL 8.x para aproveitar seu desempenho, segurança e confiabilidade aprimorados. Embora o MySQL 5.7 ainda seja compatível com o Magento 2.4.x, o MySQL 5.6 não é mais suportado. Não é possível hospedar a 2.4.x com um banco de dados MySQL 5.6. Consulte [MySQL](../../../installation/prerequisites/database/mysql.md).

* **Remoção do mecanismo de pesquisa do catálogo MySQL**. O mecanismo de pesquisa MySQL foi removido de 2.4.0 e substituído como o mecanismo de pesquisa padrão por Elasticsearch. O Elasticsearch oferece recursos de pesquisa superiores, bem como otimizações no desempenho do catálogo.  Todos os comerciantes devem ter o Elasticsearch para instalar e implantar a versão 2.4.0. Consulte [Verifique se o Elasticsearch está instalado](https://experienceleague.adobe.com/docs/commerce-operations/upgrade-guide/prepare/prerequisites.html).

* **Suporte para MariaDB 10.4**. O suporte para o MySQL 8.0 oferece a oportunidade para os comerciantes implantarem o MariaDB 10.4 com o Magento Open Source. Embora os comerciantes ainda possam usar o MariaDB 10.2 com 2.4.0, recomendamos atualizar para o MariaDB 10.4 para melhorar o desempenho e a confiabilidade. Os aplicativos MariaDB 10.0 e 10.1 não são mais compatíveis (como resultado da remoção do suporte ao MySQL 5.6 nesta versão).

* **Migração de dependências da Zend Framework para a [Projeto Laminas](https://getlaminas.org/about/foundation)** para refletir a transição do Zend Framework para o Projeto Laminas da Fundação Linux. O Zend Framework foi substituído. Consulte a [Migração do Zend Framework para o Projeto Laminas](https://community.magento.com/t5/Magento-DevBlog/Migration-of-Zend-Framework-to-the-Laminas-Project/ba-p/443251) Publicação DevBlog.

* **Descomposição de Controladores** O permite que os desenvolvedores de extensão implementem a ActionInterface diretamente sem classes de &quot;supertipo de camada&quot;. Consulte a [Descomposição de Controladores](https://community.magento.com/t5/Magento-DevBlog/Decomposition-of-Magento-Controllers/ba-p/430883) Publicação DevBlog. _Aprimoramento iniciado pelo Vinai Kopp em solicitação de pull [16268](https://github.com/magento/magento2/pull/16268) e finalizada por Lukasz Bajsarowicz em pedido de pull [26778](https://github.com/magento/magento2/pull/26778)_. [GitHub-9582](https://github.com/magento/magento2/issues/9582)

* **Remoção da integração central do código de proteção da fraude Signifyd**. Esse recurso principal não é mais suportado. Os comerciantes devem migrar para o [Fraude Signifyd e extensão de Proteção contra Chargeback](https://marketplace.magento.com/signifyd-module-connect.html) que está disponível no Commerce Marketplace.

* O **o módulo Braintree principal foi removido da base de código**. O módulo Braintree Payments agora fornece o mesmo conjunto de recursos. Consulte [Braintree Payments](https://marketplace.magento.com/paypal-module-braintree.html).

* O navegador Internet Explorer 11.x não é mais compatível.

### Melhorias na infraestrutura

Esta versão contém aprimoramentos à qualidade principal, que melhoram a qualidade da Estrutura e dos seguintes módulos: Conta do cliente, Catálogo, CMS, Importação, Carrinho e Check-out e B2B.

* **Remoção da integração fundamental dos métodos de pagamento de terceiros**. Com esta versão, a integração do método de pagamento Authorize.Net foi removida do código principal. Os comerciantes devem migrar para a extensão oficial que está disponível no Commerce Marketplace. Consulte a [Substituição de integrações de pagamento](https://community.magento.com/t5/Magento-DevBlog/Deprecation-of-Magento-core-payment-integrations/ba-p/426445) publicação do devblog. <!-- MC-29029-->

* **Suporte para pesquisa de palavras parciais para o Elasticsearch (novo mecanismo de pesquisa padrão)**. O Elasticsearch agora oferece suporte ao uso de palavras parciais em termos de pesquisa para nomes de produtos e SKUs ao usar uma pesquisa rápida. Esse recurso era compatível com o mecanismo de pesquisa MySQL, que foi substituído e substituído pelo Elasticsearch nesta versão.

* **Atualização do SDK do JavaScript do PayPal**. Migramos a integração do PayPal Express Checkout para o SDK JavaScript do PayPal mais recente, um SDK que coleta e transmite automaticamente os parâmetros de risco necessários para o PayPal. O comportamento do método de pagamento do PayPal Express Checkout permanece inalterado. No entanto, a atualização desse SDK para a versão mais recente permite que os comerciantes acessem os recursos e as melhorias de segurança mais recentes. <!-- MC-30962-->

* **Substituição e remoção do Assistente de configuração da Web**. Você deve usar a linha de comando para instalar ou atualizar a versão 2.4.0. Consulte [Instalar o Magento](../../../installation/advanced.md).

* **Plug-in de atualização do Composer**. O plug-in do Composer simplifica o processo de atualização, solucionando alterações que devem ser feitas no projeto raiz `composer.json` antes de atualizar para uma nova versão. Esse plug-in protege contra personalizações de substituição. Consulte [Atualizar usando o plug-in raiz do compositor](https://experienceleague.adobe.com/docs/commerce-operations/upgrade-guide/implementation/perform-upgrade.html).

* **Compras assistidas por vendedores**. Esse recurso permite que os comerciantes visualizem a loja em nome de seus clientes. Os clientes optam por permitir acesso à loja para suas contas. Esse recurso desenvolvido pela comunidade inclui uma extensão original desenvolvida por [MAGEFAN](https://magefan.com/). Consulte [Compras assistidas pelo vendedor](https://docs.magento.com/user-guide/configuration/customers/login-as-customer.html). Os recursos incluem:

   * ACL para controlar quais administradores podem fazer logon em contas de clientes podem ser configuradas por site
   * Compatibilidade com vários sites e escopos de conta do cliente
   * Os pedidos feitos em nome dos clientes são registrados na loja e no Administrador
   * Todas as sessões são destruídas após o logout do administrador e os administradores não podem acessar senhas do cliente.

### Melhorias de desempenho

* **Melhorias na lógica de invalidação da seção de dados do cliente**. Esta versão apresenta uma nova maneira de invalidar todos os dados das seções do cliente, o que evita um problema conhecido com o armazenamento local quando personalizado `sections.xml` as invalidações estão ativas.  (Anteriormente, o conteúdo privado (armazenamento local) não era preenchido corretamente quando você tinha um *etc/frontend/sections.xml* com invalidações de ação.) Consulte [Conteúdo privado](https://developer.adobe.com/commerce/php/development/cache/page/private-content/).

* **Várias otimizações para o desempenho do Redis**. As melhorias minimizam o número de consultas ao Redis realizadas em cada solicitação. Essas otimizações incluem:

   * Diminuição do tamanho das transferências de dados de rede entre Redis e Magento

   * Redução do consumo de Redis de ciclos da CPU, melhorando a capacidade do adaptador de determinar automaticamente o que precisa ser carregado

   * Redução das condições de corrida em operações de gravação de Redis
   Consulte [Use o Redis para a página e o cache padrão](../../../configuration/cache/redis-pg-cache.md) e [Configurar armazenamento em cache](../../../configuration/cache/caching-overview.md).

* **Melhoria no cache de resultados de consultas SQL para tabelas de inventário**. Esses aprimoramentos incluem:

   * Armazenamento de consultas SQL em cache para o `inventory_stock_sales_channel` tabela (1 query em vez de 16)

   * Armazenamento em cache do resultado de consultas para o `inventory_stock` tabela (1 query em vez de 16)

* **Melhoria de até 25 a 30% no desempenho do complemento para o carrinho do Quick Order**.

* Os comerciantes agora podem usar [carregamento lento](https://en.wikipedia.org/wiki/Lazy_loading) para carregar imagens.

## Problemas conhecidos — Geral

**Problema**: Anomalias em mensagens de erro de vitrine ocorrem em implantações em que o PHP 7.4.2 está instalado. Quando a 2.4.0 é implantada com o PHP 7.4.2, os símbolos de espaço nas mensagens de erro da frente de loja são substituídos por caracteres de adição (+).  Este bug é nativo do PHP 7.4.2 e não pode ser corrigido pelo Magento Open Source. **Solução alternativa**: Recomendamos o uso de outras versões do PHP 7.4.x. ## Problemas conhecidos — VBE

### Amazon Pay

* **Problema**: Os métodos de pagamento estão ausentes no fluxo de trabalho de finalização quando um cliente clica **Retornar ao check-out padrão** durante o check-out com Amazon Pay. **Solução alternativa**: Atualize a página de checkout para exibir os métodos ausentes.  <!-- BUNDLE-2661-->

* **Problema**: O aplicativo exibe dois métodos de Pagamento Amazon idênticos na lista suspensa Método de Pagamento, que está disponível ao criar uma nova regra de preço do carrinho.  <!-- BUNDLE-2685-->

### Braintree

* **Problema**: O aplicativo exibe um erro quando um administrador tenta acessar a página Relatório de Liquidação de Braintree (**Administrador** > **Relatórios** ). No momento, o aplicativo exibe esta mensagem: `An error has happened during application run. See exception log for details`. Consulte [Falha no carregamento do relatório de liquidação de Braintree](https://support.magento.com/hc/en-us/articles/360046798251) Artigo da Base de conhecimento. O **BUNDLE-2683_SettlementReport.patch** O hotfix para esse problema agora está disponível em [Versões](https://magento.com/tech-resources/download). <!-- BUNDLE-2683-->

* **Problema**: Os comerciantes não podem criar faturas parciais para pedidos em implantações nas quais Venmo está ativado e o **Habilitar Cofre para Pagamentos com Cartão** está desativada. Atualmente, o aplicativo exibe este erro: `The "vault_capture" command doesn't exist. Verify the command and try again`.  <!-- BUNDLE-2647-->

* **Problema**: O aplicativo exibe dois métodos PayPal idênticos na lista suspensa Método de Pagamento que está disponível ao criar uma nova regra de preço do carrinho.   <!-- BUNDLE-2686-->

* **Problema**: Erro do aplicativo ao abrir **Vendas** > **Braintree Virtual Terminal**. Embora o formulário contenha elementos de interface do usuário corrompidos, ele ainda aceita pagamentos. **Solução alternativa**: Salve as credenciais de Braintree corretas, que corrigirão os campos de entrada recolhidos. Consulte [A página Braintree Virtual Terminal está corrompida](https://support.magento.com/hc/en-us/articles/360046800071)  Artigo da Base de conhecimento. ### Klarna

* **Problema**: Sair do **Tema de Design** campo em branco em Mensagens no local de Klarna (**Administrador**  >  **Lojas**  >  **Configuração**  > **Configurações de Klarna**) resulta em uma página em branco.  **Solução alternativa**: Selecione um tema de design e salve.  <!-- BUNDLE-2623-->

### Vertex

* **Problema**: o aplicativo exibe um  **Verificação de endereço** na seção de envio do fluxo de trabalho de check-out, mesmo quando a validação de endereço estiver desativada. O validador de endereços não verifica novamente o endereço atualizado e continua exibindo uma mensagem que indica que o endereço é inválido quando um cliente insere um endereço correto após inserir um endereço incorreto pela primeira vez, mesmo quando a validação de endereço está ativada. **Solução alternativa**: Desative a validação do endereço de vértice. <!-- BUNDLE-2604-->

## Kits de ferramentas de migração

A Ferramenta de migração de dados ajuda a transferir os dados de armazenamento do Magento 1.x existentes para o Magento 2.x. Essa interface de linha de comando inclui funções de verificação, rastreamento de progresso, registro e teste. Para obter instruções de instalação, consulte [Instalar a ferramenta Migração de dados](../../../tools/data-migration-tool/install.md). Considere explorar ou contribuir para a variável [Repositório de migração de dados](https://github.com/magento/data-migration-tool).

O [Kit de ferramentas de migração de código](https://github.com/magento/code-migration) ajuda a transferir extensões e personalizações de loja do Magento 1.x existentes para o Magento 2.x. A interface da linha de comando inclui scripts para conversão de módulos e layouts do Magento 1.x.
